# AGENTS.md

## What this repo is

**El Corte** — a barbershop landing page (Spanish). React 18 + TypeScript + Vite, frontend-only **Progressive Web App**.
Sections: full-viewport hero, numbered services grid, booking form, footer.

The booking form POSTs to `/api/bookings` — **this endpoint does not exist in this repo**. The backend is out of scope; the form will always get a network error in dev/preview.

---

## Commands

```sh
npm run dev        # dev server → http://localhost:3000
npm run build      # production build → dist/
npm run preview    # preview build on port 4173
npm run test       # vitest (watch mode)
npm run test -- --run  # single-pass, used in CI
npm run format     # prettier over src/**
```

**Do not use `npm run lint` or `npm run ci`** — both fail because `eslint-plugin-jsx-a11y` is declared in `.eslintrc.cjs` but is missing from `package.json` and not installed. This is a pre-existing repo defect, not introduced by code changes. CI will fail on the lint step until the package is installed:
```sh
npm install eslint-plugin-jsx-a11y --save-dev
```

## Architecture

```
src/
  App.tsx                        # owns all static data; injects via props
  components/                    # layout/presentational
    Header.tsx                   # scroll-aware, nav driven by NAV_ITEMS[]
    Hero.tsx                     # full-viewport typographic, no image
    Services.tsx                 # accepts Service[] prop
    Footer.tsx                   # accepts socialLinks[] prop
  features/booking/
    BookingForm.tsx              # form state + submit; local FormField component
    booking.types.ts             # BookingRequest type
  services/api.ts                # postJSON<TReq, TRes> + ApiResponse<T> union type
  utils/validate.ts              # validateBooking(input) → string[]
  styles/index.css               # single file; all styles; CSS custom properties design system
```

Static content (services list, nav items, phone, social links) lives in **`App.tsx`** — change it there, not inside individual components.

---

## Styling

Single `src/styles/index.css` — no CSS modules, no styled-components. All design tokens are CSS custom properties defined in `:root`. Fonts are Epilogue (sans-serif), loaded via `<link>` in `index.html`.

When adding new components, follow the existing BEM-ish flat class naming and use `var(--accent)`, `var(--border)`, `var(--muted)` etc. from the design system rather than raw color values.

## PWA (Progressive Web App)

**Functional PWA** with offline support via `vite-plugin-pwa`:

- **Service Worker**: Auto-generated by Workbox, auto-updates on new deploys
- **Manifest**: `public/manifest.json` — app name, icons, theme colors
- **Offline fallback**: `public/offline.html` — minimalist "Sin conexión" page shown when navigation fails without network
- **Cache strategy**: 
  - Static assets (CSS, JS, fonts): precached on install
  - Google Fonts: CacheFirst with 1-year expiration
  - API calls (`/api/*`): NetworkOnly (never cached)

**Testing PWA locally**:
```sh
npm run build
npm run preview
# Open http://localhost:4173 in Chrome/Edge
# DevTools → Application → Service Workers (verify registered)
# DevTools → Network → Offline checkbox (test offline fallback)
```

**Install prompt**: On mobile (Chrome/Edge), tap "Add to Home Screen". On desktop, look for install icon in address bar.

**Icons**: Currently using `favicon.svg` as icon source. For production, generate proper PNG icons (192x192, 512x512) and update `vite.config.ts` manifest.icons.

## TypeScript quirks

- `strict: true` + `noUncheckedIndexedAccess: true` — array index access returns `T | undefined`
- `baseUrl: "src"` — imports can use `@/components/Foo` or `components/Foo` (both resolve)
- `moduleResolution: "Bundler"` — Vite-aware, not Node16

## Testing

- Vitest + jsdom + `@testing-library/react`; tests live in `test/unit/`
- Tests use `getByLabelText` — form inputs **must be associated with `<label>` elements** (or `aria-labelledby`). Using `<span>` instead of `<label>` silently breaks tests. `FormField` in `BookingForm.tsx` must wire labels properly.
- Mock pattern: `vi.spyOn(api, 'postJSON').mockResolvedValue(...)` — import the module, spy on the export
- Run a single test file: `npm run test -- test/unit/BookingForm.test.tsx --run`

## Vite config note

`vite.config.ts` uses an async dynamic import for the React plugin (unusual but intentional — avoids `require()` issues). Follow the same async pattern if adding more plugins:
```ts
const myPlugin = (await import('vite-plugin-x')).default;
```
